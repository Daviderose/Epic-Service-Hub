Epic Service Hub API (0.0.1)

Introduction

Welcome to the Epic Service Hub API. A robust data service that is secure, reliable and fast. The Epic Service Hub API takes the difficult work of accessing all sorts of patient data from Epic without the need to write elaborate and extensive SQL queries.

The Epic Service Hub API is designed for developers, engineers, or anyone else who is comfortable creating custom-coded solutions or integrating with RESTful APIs. If you would like to learn more about REST before using the API, check out our Introduction to REST guide.

Getting Started

Before You Start

If you are interested in integrating your app or software platform with Epic Service Hub, you will first need to register and authenticate your service with our API. Also you may find it helpful to familiarize yourself with our terms of use.

Once you have a valid API key, you can view and test out each endpoint of our API via the corresponding swagger links indicated by the swagger icon. These icons are both located to the right of each quick link and next to the header of each service's article.

When you click into a swagger document you will see detailed information on each service's endpoint as well as a button in the upper-right corner that says "Try it out". Clicking on this will allow you to test out the endpoint after filling out the appropriate information for the headers/parameters.

Finally, after clicking "Try it out" you simply add the required information and then click "Execute" below. Your response body will pop up below that with a corresponding response code.

Services

The Epic Service Hub API provides several services for accessing patient and clinical information from Epic. Each service is divided up by functionality. Below is a list of our services along with a brief description about what they do and how to access them.

Patient Services: Understanding The Big Picture

Before going into detail about each service, it may be helpful to understand how some of these services work together. For instance, The patient info service, the patient medication service and the patient billing service can all be called separately or they can all work together in a single call.

First, the patient info service is called to pull in the demographic information by searching a patient's phone number, if a patient is found, then the service will call both the medication and billing services using the optional MRN and Epic Instance that may be pulled in from the patient's demographics. If any one of the three services don't find a result, a message will display that no results were found.

Appointment Service

The Appointment Service API provides access to any upcoming appointments that a patient might have. A user will be returned the appointment start time, the appointment date, the name of the provider, and the type of visit.

Care Team Service

The Care Team Service API provides access to physicians that are part of a patient's care team. The user will be returned information about both the patient and the physicians that are a part of the patient's care team.

Contract Service

The Contract Service API provides access to contracts between Physicians and locations that they are contracted to practice at. Contracts can be pulled up one of two ways, either by providing a search term (with or without a relevant category like "providers") or bringing up a specific contract by entering in the contract id.

Department Service

The Department Service API provides a secure way to geographically search for Epic departments. These departments can be accessed either by using coordinates (lattitude and longitude) or by searching a procedure. The latter will return departments that provide services for that particular procedure.

Knowledge Service

The Knowledge Service API provides access to Knowledge articles for procedures, instructions and insurance plans. There are six different API calls which can be used and are divided by functionality. Knowledge can be searched both generally and specifically by type. It also searches through indexed data which makes searching not only quick, but it can be used with typeahead functionality.

Map Service

The Map Service API provides a secure way to search for Map Locations by either coordinates(lat and long bounds) or by procedure. When searching by procedure, only locations that perform that procedure are returned. Map locations are created from Epic Departments and are used to create locations to be displayed on google maps as pins. For more information on the google maps API click here.

Patient Billing Service

The Patient Billing Service API provides a secure way to search for a patient's billing information. The API searches based on a patient's phone number and returns the patient's outstanding balance(if any) and the clinic location where the balance is due.

Patient Health Maintenance Service

The Patient Health Maintenance Service API provides access to a patient's upcoming maintenance appointments. This API takes in a patient MRN and a location and returns the upcoming appointments for that location.

Patient Info Service

The Patient Info Service API provides access to general patient data from Epic. This service allows a user to search patients by three different methods. Either by the patient's Name, Phone Number or by their MRN.

Patient Insurance Service

The Patient Insurance Service API provides access to a patient's insurance plans and coverages. The search is performed by providing the patient's phone number.

Patient Medication Service

The Patient Medication API provides access to patient medications from Epic. This service allows a user to view a particular patient's medications by using their MRN as the patient id.

Patient Notification Service

The Patient Notification API shows a patient's preferences when it comes to being notified.

Patient Referral Service

The Patient Refferal Service API provides access to a patient's referrals.

Pharmacy Service

The Pharmacy Service API provides access to the Pharmacies that the patient prefers to use.

What is REST?

The Epic Service Hub API could best be described as "RESTful." "REST" stands for Representational State Transfer. It's an architectural style that's an alternative to RPC or SOAP web services.

While there's no official REST standard, there are common approaches and best practices used across the engineering community that help define how RESTful APIs should work. For example, most RESTful APIs follow six specific constraints or design rules.

The Epic Service Hub API follows these practices and uses the standard HTTP methods of GET and POST.

RESTful HTTP Methods

You may see these standard HTTP methods referred to as CRUD, or Create, Read, Update, Delete. Although CRUD has roots in database operations, you can also map those operations to the standard HTTP methods. For example, use a POST request to create a new resource, a GET request to read or retrieve a resource, PATCH request to edit a resource, and a DELETE request to delete a resource.

REST Methods:

GET- Retrieve the URIs of the member resources of the collection resource in the response body.

POST- Create a member resource in the collection resource using the instructions in the request body. The URI of the created member resource is automatically assigned and returned in the response Location header field.

PUT- Replace all the representations of the member resources of the collection resource with the representation in the request body, or create the collection resource if it does not exist.

PATCH- Update all the representations of the member resources of the collection resource using the instructions in the request body, or may create the collection resource if it does not exist.

DELETE- Delete all the representations of the member resources of the collection resource.

Best Practices

Don't Forget Error Handling

When you use any API, keep in mind that errors and exceptions will sometimes happen. For example, you might experience a connection problem between your servers and the Epic Service Hub, or there sometimes may be a rare outage. To make sure that your integration is as reliable as it can be, you should always watch for errors and exceptions.

We also recommend that you async your calls to the API as a background job, instead of making your users wait for a response.

Make Specific Requests

Sometimes, the API will return values that are large and slow to calculate. After you know what data you need, use our partial response capabilities to request only what's essential to you. Often, when you make fewer requests, you will generate quicker responses.

Another way to reduce response time is to make fewer calls. Calls that are too frequent can slow your application, and may also receive unwanted attention from our operations team. For example, you probably don't need to query every single care provider for every hour for several days, because this data will only change once per day. To reduce calls further, we recommend you sync only new data, and cache data that's unlikely to change.
